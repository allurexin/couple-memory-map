# 情侣美食记忆地图设计

## 概述

做一个移动端优先的网页 MVP，用来保存两个人一起吃过的食物、去过的店和相关回忆，并把这些记忆落实到真实地图上。

第一版不是纯演示原型，而是一个能真实使用的小产品：

- 入口以地图为主。
- 先做网页应用，之后再考虑微信小程序或原生 App。
- 两个人分别注册登录，再绑定到同一个情侣空间。
- 使用场景以中国大陆为主，优先接入高德地图或腾讯地图这类中文地图和地点搜索服务。
- 内容以美食记忆为核心。
- 使用真实后端、共享数据和照片存储。

## 目标

- 两个人可以分别创建账号，并绑定到同一个情侣空间。
- 登录后打开的是两个人共同的美食记忆地图。
- 可以通过地点搜索添加记忆，搜索不到时可以在地图上手动放点。
- 每条记忆可以保存店名或地点名、菜品、日期、评分、照片、备注，以及是否想再去。
- 情侣空间内的任意一方都可以查看和编辑共同记忆。
- 第一版范围足够小，能尽快做出可验证、可使用的版本。

## 不做的内容

第一版不做：

- 公开分享。
- 社交动态。
- 聊天。
- 评论区。
- 纪念日提醒。
- 路线回放。
- 餐厅推荐。
- 多人空间。
- 复杂相册排版。
- 管理后台。

## 产品范围

### 登录和绑定

用户分别注册和登录。登录后，用户可以创建一个情侣空间，或者通过对方生成的绑定码加入已有情侣空间。

如果用户还没有绑定情侣空间，应用展示引导空状态，提示创建空间或加入对方空间。

### 地图首页

绑定完成后，地图是主页面。地图首页展示：

- 当前情侣空间的所有美食记忆点。
- 顶部餐厅或地点搜索框。
- 地图标记，用颜色或小图标体现评分、想再去、一般或踩雷等状态。
- 底部抽屉，展示最近添加的记忆和筛选入口。

如果还没有任何记忆，首页引导用户添加第一家店。

### 添加记忆

常规流程从搜索餐厅或地点开始。用户选中地图服务返回的 POI 后，进入记忆表单。

如果搜索不到地点，用户可以切换到手动模式，在地图上放一个点，然后填写同一套表单。

表单包含：

- 店名或地点名。
- 经纬度。
- 城市。
- 地图 POI ID，有则保存，没有则为空。
- 日期。
- 菜品名，可以有多个。
- 评分。
- 是否想再去。
- 照片。
- 备注。

照片上传失败不能丢失文字内容。应用应先保存文字和地点信息，再提示照片可重试上传。

### 记忆详情

点击地图标记后，打开记忆详情卡片。详情卡展示：

- 照片。
- 店名或地点名。
- 菜品名。
- 评分。
- 是否想再去。
- 日期。
- 备注。
- 创建人。
- 最后更新时间。
- 编辑入口。

情侣空间内的两位成员都可以编辑已有记忆。

### 列表和筛选

底部抽屉支持筛选：

- 城市。
- 评分。
- 是否想再去。
- 菜品关键词。
- 创建人。

筛选保持轻量，作为地图的一部分出现，不做成复杂独立页面。

## 数据模型

### User 用户

表示一个账号。

字段：

- `id`
- `email` 或手机号等登录标识
- `displayName`
- `avatarUrl`
- `createdAt`
- `updatedAt`

### CoupleSpace 情侣空间

表示两个人的私密共享空间。

字段：

- `id`
- `name`
- `memberIds`
- `bindingCode`
- `createdBy`
- `createdAt`
- `updatedAt`

第一版假设一个情侣空间最多两位成员。

### Memory 记忆点

表示一个落在地图上的美食记忆。

字段：

- `id`
- `coupleSpaceId`
- `placeName`
- `poiId`
- `latitude`
- `longitude`
- `city`
- `memoryDate`
- `rating`
- `revisitStatus`
- `notes`
- `createdBy`
- `updatedBy`
- `deletedAt`
- `createdAt`
- `updatedAt`

删除使用软删除，通过 `deletedAt` 标记，避免误删后完全找不回。

### FoodItem 菜品

表示一条记忆里的一个菜品或食物。

字段：

- `id`
- `memoryId`
- `name`
- `createdAt`

### Photo 照片

表示一张上传到记忆点的照片。

字段：

- `id`
- `memoryId`
- `storagePath`
- `thumbnailUrl`
- `originalUrl`
- `uploadedBy`
- `createdAt`

## 权限

用户只能访问自己绑定的情侣空间。

规则：

- 用户必须登录后才能进入应用。
- 只有当用户 ID 属于 `CoupleSpace.memberIds` 时，才能读取该空间的记忆。
- 用户只能在自己的情侣空间里创建记忆。
- 用户只能编辑自己情侣空间里的记忆。
- 用户只能软删除自己情侣空间里的记忆。
- 绑定码只能用于加入未满员的情侣空间。

## 同步逻辑

第一版不需要做复杂的实时协同编辑。

预期行为：

- 创建或编辑记忆的人立即看到更新。
- 另一位用户刷新页面、重新打开页面或切回页面时，能看到最新数据。
- 应用在创建、编辑、删除、登录、绑定和页面重新获得焦点时拉取最新记忆数据。

这样实现更稳定，同时已经足够像一个真实的共享产品。

## 地图集成

第一版面向中国大陆使用，优先接入高德地图或腾讯地图，用于：

- 地图渲染。
- POI 地点搜索。
- 地点选择。
- 经纬度获取。

地图服务异常时需要降级：

- 已有记忆仍然可以在列表里查看。
- POI 搜索失败时给出清晰提示。
- 搜索不可用时仍允许手动放点。
- 如果没有 POI ID，保存用户填写的店名和手动选择的经纬度。

## 异常和空状态

需要覆盖：

- 未登录。
- 已登录但未绑定情侣空间。
- 情侣空间没有任何记忆。
- 地点搜索无结果。
- 地图服务不可用。
- 照片上传失败。
- 网络请求失败。
- 绑定码无效。
- 绑定码对应的情侣空间已满。
- 用户尝试访问别人的情侣空间。

当照片上传或地图搜索失败时，应用不能丢失用户已经输入的文字内容。

## 测试策略

### 核心流程测试

需要测试：

- 注册和登录。
- 创建情侣空间。
- 通过绑定码加入情侣空间。
- 通过 POI 搜索添加记忆。
- 通过手动放点添加记忆。
- 上传照片。
- 从地图标记打开记忆详情。
- 编辑记忆。
- 软删除记忆。
- 筛选记忆。

### 权限测试

需要测试：

- 用户不能读取其他情侣空间。
- 用户不能在其他情侣空间创建、编辑或删除记忆。
- 已满员的情侣空间不能再通过绑定码加入第三个人。

### 失败场景测试

需要测试：

- POI 搜索失败。
- 地图加载失败。
- 照片上传失败。
- 创建或编辑时网络失败。
- 刷新页面和页面重新获得焦点后的数据更新。

## 验收标准

第一版完成时应满足：

- 两个用户可以分别注册和登录。
- 一个用户可以创建情侣空间。
- 第二个用户可以绑定到同一个情侣空间。
- 两个用户绑定后能看到同一张地图。
- 任意一方可以通过搜索地点添加美食记忆。
- 任意一方可以通过手动放点添加美食记忆。
- 一条记忆可以包含菜品、评分、是否想再去、备注、日期和照片。
- 另一方刷新或重新进入应用后能看到新增记忆。
- 点击地图标记能打开正确的记忆详情。
- 任意一方可以编辑情侣空间里的已有记忆。
- 筛选可以按城市、评分、是否想再去、菜品关键词和创建人生效。
- 用户不能访问其他情侣空间的记忆。

## 实现阶段再决定的事项

这些属于实现选型，不影响已经确认的产品范围：

- 前端框架。
- 后端平台。
- 地图服务最终选择高德还是腾讯。
- 照片存储服务。

这些会在实现计划阶段根据 API Key、部署便利性和本地开发环境来决定。
